Refueler — Cashu, GDPR & C-Suite Briefing Document

Cashu ecash — end-to-end in the Refueler order flow

Cashu is a Chaumian blind-signature ecash protocol. The core privacy guarantee is that the mint signs tokens without ever seeing which token belongs to which user. Tokens are bearer instruments — whoever holds the proof, owns the sats.

The six-step flow

1
Mint funding
Refueler funds its Cashu mint (e.g. mint.minibits.cash or a self-hosted Nutshell instance) via Lightning. This is an operational float — sats held in reserve to back all ecash tokens Refueler will ever issue. This is Refueler's custodial exposure; the mint holds the keys to this float.
2
User completes an order — reward trigger
On collection confirmation (NFC tap or QR scan), the Refueler backend calculates the sats reward for this order. No token exists yet.
3
Blinded token request (blind signature)
The user's wallet (Minibits) generates a random secret, blinds it cryptographically, and sends only the blinded value to the mint. The mint never sees the original secret — this is the Chaumian blinding step. The mint signs the blinded value and returns the blind signature.
4
Token unblinding
The wallet unblinds the mint's signature using its original secret. The result is a valid Cashu proof (token): a pair (secret, signature) that the mint will honour. The mint has no record linking this proof to the user who requested it.
5
Token stored on device
The proof lives only in Minibits on the user's device. Refueler's Supabase has no record of which token was issued to which user. The orders table records that a reward was issued (for HMRC compliance) but not the token value or the unblinded proof.
6
Redemption
When the user wants to move sats out (to Lightning, or to spend elsewhere), they redeem the Cashu proof against the mint. The mint checks the proof hasn't been spent before (double-spend prevention via a spent token database) and pays out via Lightning. The user is unlinkable to the original issuance.
Trust model advisory: Refueler's mint is custodial. The user trusts the mint operator (Refueler, or Minibits' public mint) not to inflate the supply or rug the float. This is the primary trust risk to disclose. Enclaves (see that tab) may resolve this in future.
Lightning address — personal data status under UK GDPR
Position: Yes — a Lightning address is personal data under UK GDPR. The ICO's definition covers any identifier that can identify a person "directly or indirectly." A Lightning address (e.g. rajesh@minibits.cash) is structurally identical to an email address. It identifies a specific wallet, typically controlled by one person, and often contains a username. It must be treated as personal data from the moment it is collected.

Legal basis

Retention position — locked

Lightning address must be deleted immediately after the payment is dispatched. The payment event itself (amount, timestamp, order ID) is retained for 6 years (HMRC financial records). The address is not part of that record — it is a routing instrument only.

Concretely: the address should never be written to the orders table. It should exist only in transient server memory during the payment execution, then discarded. No column, no log entry, no backup.

Data (Use and Access) Act 2025: Royal Assent June 2025. Introduces "recognised legitimate interests" as a new legal basis — but this applies only to five specific public interest scenarios and does not expand the basis available to a commercial rewards product. Refueler's basis for the Lightning address remains Art. 6(1)(b) — contract. Monitor ICO guidance updates (final guidance expected late 2026).
Minibits onboarding — what Refueler controls vs what Minibits controls

Minibits is an independent open-source wallet. Refueler cannot modify it. The boundary of control matters for both UX design and GDPR data processor mapping.

Refueler controls

✓
Triggering the handoff
Refueler decides when to prompt: post-collection, when the user has accumulated ≥1 sat reward with no wallet linked
✓
The Cashu token payload
Refueler generates the token string. The format is a standard Cashu token (base64-encoded serialised proof). Minibits reads it; Refueler constructs it.
✓
The deep link / handoff UX
Refueler can use a cashu: URI scheme or a universal link to pass a token directly into Minibits. This creates a one-tap "open in Minibits" experience from within Refueler.
✓
Pre-handoff education copy
Refueler's screens before the handoff: what is a wallet, what are you about to receive, why it matters. Concierge register throughout.
✓
Mint selection
Refueler decides which mint backs the token. Default: Minibits' public test mint for MVP, with a path to a Refueler-operated mint post-launch.

Refueler does not control

✗
Minibits wallet setup screens
Seed phrase backup, wallet name, settings — these are Minibits' UX. Refueler can guide the user to this point; it cannot customise what happens inside Minibits.
✗
Token receipt confirmation
Refueler cannot know if the user successfully received and stored the token inside Minibits. A separate Lightning-address-based confirmation flow is the only feedback mechanism.
✗
Minibits privacy policy
Minibits' mint server processes the blind signature request. This is a separate data controller relationship. Refueler's privacy policy should disclaim the Minibits/mint relationship and link to theirs.
✗
Token recovery if device lost
Once the token is in Minibits, only Minibits' seed-phrase recovery applies. Refueler has no ability to reissue a lost token — this must be disclosed at onboarding.

Sats issuance — when does the token leave the mint and land in the wallet?

The issuance moment is important for both UX design and HMRC records. There are two distinct events, and they must not be conflated.

A
Reward committed (Refueler's obligation created)
This happens at collection confirmation — when the user taps to collect and the order status transitions to "collected." At this point Refueler's ledger records: order ID, sats amount, timestamp. The token does not yet exist in the user's wallet. This is the HMRC-recordable event.
B
Token minted and delivered (user's possession)
This happens when the blind-signature exchange completes — either synchronously at collection (if the user has Minibits open and linked) or asynchronously (the token is queued and delivered when the user next opens the app). The token lands in the user's device storage. Refueler no longer holds or knows the token value.

The orders table does NOT contain: the Cashu token proof, the blinded secret, the user's Lightning address, or any location data.

Orders table — what it must never contain
Zero-location guarantee: No location column exists or will ever be added to the orders table without a signed GDPR review. The geofence fires on-device only. The only payload sent to the server on order confirmation is: item + vendor ID + session token. This is an architectural guarantee, not a policy.

GDPR disclosures — Cashu/ecash reward processing

Two disclosures are required. The first covers the ecash reward processing at onboarding (analogous to the geofence disclosure from Session 9). The second is the wallet handoff disclosure shown at the moment the user first links or creates a wallet.

Disclosure 1 — ecash reward processing (onboarding opt-in)

"Your sats rewards are issued as Cashu ecash tokens — a form of digital cash backed by Bitcoin. When you collect an order, Refueler records that a reward was earned. The token itself is generated using a blind signature process: your wallet creates it in a way that means we cannot see which token we signed. The token lives on your device only. We never see your balance. If you link a Lightning address to receive sats directly, we use it once to send the payment and delete it immediately afterwards. We never store your Lightning address."

Legal basis: Art. 6(1)(b) — contract performance. Processing: reward issuance, payment dispatch. Retention: reward event record (amount + timestamp) — 6 years HMRC. Token: on device, not transmitted. Lightning address: transient, deleted post-payment.

Disclosure 2 — wallet handoff (first wallet link screen)

"To receive your sats, you'll need an ecash wallet. We'll hand your reward to the wallet you choose — after that, it belongs to you and we cannot access, reverse, or recover it. If you use Minibits, your wallet is managed by Minibits, not Refueler. You can find their privacy policy at minibits.cash. Your sats are real Bitcoin. Keep your wallet recovery phrase somewhere safe — it's the only way to recover your balance if you lose your phone."

This disclosure covers: third-party data controller (Minibits), irrevocability of token transfer, user responsibility for wallet backup. Not a consent trigger — informational notice under Art. 13/14 transparency requirements.

Disclosure 3 — loyalty track users (non-wallet, stamp scheme)

"Your loyalty stamps are stored in Refueler's system, linked to your account. They're not Bitcoin — they're a count of your purchases with this venue. We keep this record for as long as you have an active account, or until you ask us to delete it. You can switch to earning real Bitcoin sats at any time in your wallet settings."

Legal basis: Art. 6(1)(b) — contract performance. Retention: active account + 30 days post-deletion request. This is a separate processing activity from sats issuance and requires its own entry in the Record of Processing Activities (RoPA).

Enclaves — the most significant recent development in Cashu

Breaking — announced May 1, 2026 by Calle (Cashu creator)

Non-custodial Cashu ecash mints running inside hardware enclaves. The mint operator cannot access the Bitcoin private keys. They cannot inflate the ecash supply. This is the development that could change Refueler's entire trust model.

What a hardware enclave is

A hardware enclave is an isolated compute environment — a sealed box inside a server — where code runs in a way that even the server's administrator cannot inspect the memory, extract keys, or tamper with the execution. AWS Nitro Enclaves is the most widely deployed implementation. ACINQ (the Phoenix Wallet company) already uses this architecture to protect over $100M in a Lightning node.

The key properties for Refueler: the Bitcoin private keys are generated inside the enclave and never leave it. The mint operator (Refueler) cannot access them. If the server is compromised, the attacker still cannot steal the Bitcoin. Transparent logs prove what code is running.

Why this matters for Refueler — three implications

1
Regulatory / custodial status
Calle's argument: if the operator cannot access the Bitcoin, they are not a custodian under existing law. If this holds, Refueler could operate its own mint without Money Transmitter or e-money licence exposure. This requires UK-specific legal analysis — the FCA's perimeter guidance on crypto assets is different from US law. Flag for the CTO and CPO/DPO to investigate with external counsel.
2
Security model
Even if Refueler's backend is fully compromised, the user's sats cannot be stolen. This is a material security improvement over the current model (mint on a regular server). The architecture is the same approach ACINQ uses for Phoenix Wallet — battle-tested at scale.
3
Brand and investor narrative
Refueler's mint, built on enclave architecture, is verifiably non-custodial. This is a legitimate differentiator from every ad-tech loyalty scheme and every crypto rewards product that holds user funds. It is also the final answer to the trust model question that Bitcoin-native users will ask. The copy writes itself: "We cannot steal your sats. The hardware makes it impossible."
Timeline: Enclave support for Cashu was announced May 2026 and is not yet production-ready. Refueler should monitor the Cashu GitHub (Nutshell) for enclave branch development. This is a Session 11+ roadmap item — not a blocker for MVP but worth scoping now so the architecture leaves room for it.

Why it matters: if sats rewards above a threshold are reportable, Refueler needs a P11D-equivalent process. HMRC's cryptoasset guidance is evolving rapidly — this needs a dated answer against current HMRC crypto guidance.

Why it matters: the public Minibits mint is suitable for MVP but is a third-party dependency with no SLA. A self-hosted mint is the production architecture. This question scopes the migration cost and timeline.
CTO
When the Cashu blind-signature exchange completes between Refueler's mint and the user's Minibits wallet, what server-side logs are generated by Nutshell by default — and do any of those logs constitute personal data under UK GDPR that would require a retention policy?
Why it matters: mint logs could inadvertently capture IP addresses, timing patterns, or token amounts that are personal data. Default Nutshell logging behaviour must be audited and hardened before any production deployment.
CTO + CPO/DPO
If Refueler migrates from Minibits' public mint to its own Nutshell mint, does Minibits become a data processor under UK GDPR during the transition period — and what Data Processing Agreement terms are required before any user token handoff between mints?
Why it matters: inter-mint token swaps (a user's Minibits-held token being redeemed and reissued by Refueler's mint) involve both mints processing data. The legal relationship needs mapping before the migration is executed.
